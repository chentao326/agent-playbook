# 别把 Hermes 当 Claude Code 用

## —— Hermes 真实使用心得

我后来发现，Hermes 最顺手的位置，是站在微信和执行工具之间——不是又一个 coding agent，而是一个常驻入口。

---

## 一、把它想成另一个 Claude Code，就拧巴了

都能当 agent，都能读上下文、调工具。但用下来才发现：差别不在能不能，而在「站在哪里」。

**Claude Code：** 坐在项目仓库里。它擅长进项目目录，改代码、跑测试、读报错、修文件。一个本地的执行者。

**Hermes：** 坐在微信入口里。它顺手——不打开项目、不切终端，微信里发一句话，就把任务交了出去。

## 二、动作很小，差别却很大

不是在某个项目目录里跟 coding agent 对话，而是在一个常驻入口里，安排一个任务。

- **常驻入口：** Hermes 在服务器上，微信是我随手能碰到的遥控器。
- **不切工具：** 不开 IDE、不开终端、不去网页。一句话就把任务交出去。
- **状态可见：** 它会回真实情况，而不是只回一句「已完成」。

## 三、它先把一句话，整理成一个能执行的 prompt

「帮我做一篇小红书长文」—— 一句模糊需求。

Hermes 会把它整理成结构化的 prompt，包括：

- 受众：小红书读者（认真用 agent、关心怎么落地的人）
- 布局：9-12 页长图（1080×1440，标题不压边）
- 标题方向：「别把 X 当 Y 用」（反直觉、有立场、不写成评测）
- 口吻：真实使用心得（第一人称、不官方、不堆形容词）
- 避免：「全能 / 神器 / skills 卖点」（不写成产品官网）
- 交付：HTML + 分页 JPG + ZIP（顺序 01..11，可直接上传）

## 四、一句话，被拆成一张图

Hermes 不抢着写。它把任务整理好，把执行者叫起来，把上下文和限制交代清楚——然后回来收尾。

```
入口（微信） → 「帮我做一篇长图」
编排者（Hermes） → 写 prompt · 拆任务
执行者：
  - Claude Code：起草 / 改稿 / 控语气
  - Codex：SVG / TikZ / 片段
  - Scripts：导出 / 校验 / 打包
产物：分页 JPG → PDF → ZIP（可上传）
```

## 五、那些碎掉的细节，才是真正的麻烦

我最在意的不是「能不能生成内容」，而是它能不能把这些麻烦事接住。

生产清单：

- ✓ 尺寸：每页 1080 × 1440，尺寸对、不变形、可直接上传小红书。
- ✓ 安全边距：标题不压边，顶/底/左/右安全边距，文字不贴墙、不溢出。
- ✓ 页数：article → 11 P，封面+正文+封底连贯，不漏不重。
- ✓ 命名：文件名有序 01.jpg, 02.jpg ... 11.jpg，批量上传顺序不乱。
- ✓ 格式：导出可上传，JPG 体积、PDF、ZIP 三件套都打好。
- ✓ 降级：失败有降级，ZIP 发不出？换链接。图太大？降级成可下载文件。

## 六、多数 agent 停在「它生成了什么」，我在意的，是后半段

图片有没有导出？ZIP 有没有打包？顺序对不对？发得出去吗？发不出去要不要降级？

交付回执示例：

- 检查 11 张分页
- pages_jpg/01..11.jpg · 1080×1440 ✓
- long_image.pdf · 11 pages ✓
- redesign.zip · 18.4 MB ✓
- ZIP > WeChat 文件上限 ⚠
- fallback: upload, share link → link sent · expire 7d ✓
- 真实情况已回报，不只是「已完成」

## 七、它们都是 agent，差别在站的位置

都能接上下文、规则、工具。我更在意：它被放在工作流的哪个位置。

**四种 agent 的定位对比：**

| Agent | 位置 | 定位 |
|-------|------|------|
| Hermes | 微信里的常驻入口 | 把任务拆开，再把执行交给合适的工具 |
| Claude Code | 项目仓库里的执行者 | 进项目目录，改代码、跑测试、看报错、修文件 |
| Codex | 素材与片段的副手 | 需要时叫起来：SVG、TikZ、HTML 片段、跑脚本 |
| Scripts | Agent 框架与工具链 | 同一个大类里的另一种组合 |

不要把 skills 写成谁独占的卖点 —— 这是大类共有能力，差别在「放在工作流的哪个位置」。

## 八、一张图片长文，是怎么穿过这条流水的

我不会上来让一个模型一次性写文、配图、排版、打包。我先在微信里找 Hermes。

1. **YOU · 微信** —— 「帮我做一篇小红书长文」—— 一句话开始，不打开 IDE，不切终端。
2. **HERMES** —— 把模糊需求整理成 prompt：受众 / 标题 / 拆页 / 口吻 / 不出现的词 / 交付格式。
3. **CLAUDE CODE** —— 起草无图正文，控制语气、避免某些表达，整理成适合分页的结构。
4. **YOU · 微信** —— 「正文 OK，继续做长图」—— 一句话切下一段。
5. **CODEX / 脚本** —— 补素材：SVG、片段、模板填充，谁合适谁出场。
6. **HERMES** —— 回来收尾：导出图片、检查尺寸、打 ZIP、回报状态。失败就降级，不只是回「已完成」。

---

## 核心洞察

Claude Code 是很强的执行者，而 Hermes 更适合做你在微信里随手叫到的，那个任务入口。

**不是谁更强一句能说清的事。而是它被放在工作流的哪个位置。**

---

## 结语

把 Hermes 放在微信里，它不只是 agent。它是一条路：把任务送到结果。

```
微信 → Hermes → tools / executors → 结果
```
